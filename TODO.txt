6502 Simulator Project - TODO and Enhancements
===========================================

This document outlines suggested next steps, feature enhancements, and technical improvements for the 6502 simulator.

1. CPU & Architecture Support
----------------------------
*   [ ] Full 45GS02 Instruction Set: Complete missing opcodes for the MEGA65 (45GS02) processor.
    *   [X] Missing Branches (16-bit Relative): BPL, BMI, BVC, BVS, BCC, BCS, BNE, BEQ, BSR (0x13, 0x33, 0x53, 0x73, 0x93, 0xB3, 0xD3, 0xF3, 0x63) - Includes 'L' prefix support (e.g., LBVS, LBSR).
    *   [X] Missing Indirect Z-indexed: ORA, AND, EOR, ADC, STA, LDA, CMP, SBC (zp),Z (0x12, 0x32, 0x52, 0x72, 0x92, 0xB2, 0xD2, 0xF2).
    *   [X] Missing Stack Relative Indirect Y: STA (sp,s),Y (0x82), LDA (sp,s),Y (0xE2).
    *   [X] Missing JSR Indirect: JSR (abs) (0x22), JSR (abs,X) (0x23).
    *   [X] Missing Word (16-bit) Instructions: ASW, ROW, INW, DEW, PHW.
    *   [X] Missing Quad (32-bit) Instructions: Implement compound instructions using double NEG ($42 $42) prefix: LDQ, STQ, ADCQ, SBCQ, CPQ, EORQ, ANDQ, BITQ, ORQ, ASLQ, ASRQ, LSRQ, ROLQ, RORQ, INQ, DEQ.
    *   [ ] Verify/Complete: MAP (0x5C stub), NEG (0x42), RTN (0x62).
    *   [X] EOM Prefix Logic (quad-flat): execute_from_mem() now detects $42 $42 $EA $base (quad-flat) vs $42 $42 $base (quad) and sets eom_prefix=2 directly; dispatch_table_t gained quad_eom[256] for routing.
*   [ ] 45GS02 Architectural Features:
    *   16-bit Stack Pointer: Update `cpu_t` to support 16-bit `S` register and implementation of CLE/SEE (E flag).
    *   E Flag (Extended Stack): Implement bit 5 of the status register to toggle 16-bit stack.
    *   32-bit Address Support: Implement the 45GS02's ability to access up to 1MB/4GB via MAP and EOM instructions.
*   [ ] Decimal Mode Accuracy: Ensure ADC/SBC correctly handle the D flag across all CPU variants (6502 vs 65C02 differences).
*   [ ] Cycle Accuracy: Refine cycle counts for page crossings, branch success/failure, and 65CE02/45GS02 specific timing.
*   [ ] Processor-Specific Reset State: Different CPU variants have slightly different register states after RESET.
*   [ ] Complete Opcode Byte Values: Provide the correct machine code byte value for all opcodes in all processor tables (currently only partially implemented).
*   [X] Fix Branch Logic: Resolved double-increment bug in Bxx instructions for 6502 (opcodes_6502.c).
*   [ ] Fix Branch Logic: Apply similar fixes to 65C02, 65CE02, 45GS02 opcode files.

2. Memory System Enhancements
----------------------------
*   [X] Simulator Refactoring: Converted to ROM-based execution model to support loops and backward jumps.
*   [X] Self-Modifying Code (SMC): Execution model switched from static ROM dispatch to execute_from_mem(), enabling programs to write to and execute from any address — including patching opcodes, operands, and jump targets at runtime. Default program origin moved from $0800 to $0200 (conventional placement above ZP and stack). Test coverage: smc_loop, smc_patch_jump, smc_patch_opcode, smc_store_operand.
*   [X] Label Resolution: Implemented correct label resolution for relative branches (BNE, BEQ, etc.).
*   [ ] Memory Mapping (MMU): Support for banking and the MEGA65 MAP instructions.
*   [ ] I/O Region Simulation: Implement basic hardware registers (e.g., VIC-II, SID, or MEGA65 specific registers) to allow running more complex programs.
*   [ ] Read/Write Callbacks: Allow attaching functional hooks to specific memory addresses (useful for simulating hardware I/O).
*   [ ] Memory Protection: Add the ability to mark regions as Read-Only or No-Execute.

3. Debugging & Trace Features
----------------------------
*   [ ] Allow binary file retrieval of mem (eg: Bload "filename" LOAD_ADDRESS) will load the filename at the load address. 
*   [ ] Allow binary file saving of mem (eg: BSave "filename" START END ) will save the filename at address range.
*   [ ] Allow disassembly of memory (eg: D or Disasm START END) if just D then use PC. 
*   [ ] Improved Trace Formatting: Make trace output more readable and include register changes (e.g., "A: 00 -> 01").
*   [ ] Conditional Breakpoints: Support breakpoints that trigger only when a condition is met (e.g., "PC == $1234 && A == $00").
*   [ ] Reverse Debugging: Implement a limited "step back" capability by keeping a history of CPU states.
*   [ ] Stack Visualization: Add a dedicated view for the current stack contents and depth.
*   [ ] Disassembler Integration: Improve the trace and memory viewer to show disassembled instructions alongside hex values.

4. Performance & Infrastructure
------------------------------
*   [ ] JIT or Optimized Interpreter: Explore performance optimizations for long-running simulations.
*   [ ] Unit Test Suite: Create a comprehensive test suite for all opcodes, especially for edge cases like decimal mode and overflow flag.
*   [ ] Integration with External Tools: Support loading standard binary formats (e.g., .prg, .bin) and symbol files from common assemblers (ca65, acme).
*   [ ] CMake Support: Add a CMakeLists.txt for better cross-platform build support.

5. User Interface (CLI/Web/MCP)
--------------------------
*   [X] Interactive CLI Mode: A "monitor" style interface where users can type commands (step, continue, examine, modify) without restarting. (Required for MCP).
*   [X] MCP Server Implementation: Wrap the simulator in a Model Context Protocol server to expose execution control to LLMs.
*   [ ] Web-based UI: Use the existing plugin-gemini structure or similar to create a visual representation of the CPU state and memory.
*   [ ] Configuration Files: Support for .ini or .json files to specify CPU type, memory maps, and pre-loaded binaries.

6. Documentation
---------------
*   [ ] API Documentation: Document the internal structures (cpu_t, memory_t) for developers.
*   [ ] Instruction Set Reference: Generate or provide a reference for the supported opcodes of each processor type.
*   [ ] Tutorial/Examples: Add more example programs demonstrating specific processor features (e.g., 45GS02 Z register, 65CE02 indirect addressing).

7. Interface Gaps (CLI vs MCP)
-----------------------------
*   [ ] **MCP Missing Features:**
    *   Symbol Table Support: No tools to load custom symbol files or use presets (e.g., C64, Mega65).
    *   [X] Breakpoints: Implemented `set_breakpoint`, `clear_breakpoint`, `list_breakpoints`, and `run_program` tools.
    *   Memory Statistics: No equivalent to the `-s` flag.
    *   Trace: No tool to enable/disable execution tracing or set a trace file.
    *   Interrupts: No tools to trigger IRQ or NMI.
*   [ ] **CLI Missing Features:**
    *   Dynamic Code Loading: CLI requires a file path; cannot accept raw assembly string arguments (unlike MCP's `load_program`).
