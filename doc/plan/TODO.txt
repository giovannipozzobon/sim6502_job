6502 Simulator Project - TODO and Enhancements
===========================================

This document outlines suggested next steps, feature enhancements, and technical improvements for the 6502 simulator.

1. CPU & Architecture Support
----------------------------
*   [X] Full 45GS02 Instruction Set: Complete missing opcodes for the MEGA65 (45GS02) processor.
    *   [X] Missing Branches (16-bit Relative): BPL, BMI, BVC, BVS, BCC, BCS, BNE, BEQ, BSR (0x13, 0x33, 0x53, 0x73, 0x93, 0xB3, 0xD3, 0xF3, 0x63) - Includes 'L' prefix support (e.g., LBVS, LBSR).
    *   [X] Missing Indirect Z-indexed: ORA, AND, EOR, ADC, STA, LDA, CMP, SBC (zp),Z (0x12, 0x32, 0x52, 0x72, 0x92, 0xB2, 0xD2, 0xF2).
    *   [X] Missing Stack Relative Indirect Y: STA (sp,s),Y (0x82), LDA (sp,s),Y (0xE2).
    *   [X] Missing JSR Indirect: JSR (abs) (0x22), JSR (abs,X) (0x23).
    *   [X] Missing Word (16-bit) Instructions: ASW, ROW, INW, DEW, PHW.
    *   [X] Missing Quad (32-bit) Instructions: Implement compound instructions using double NEG ($42 $42) prefix: LDQ, STQ, ADCQ, SBCQ, CPQ, EORQ, ANDQ, BITQ, ORQ, ASLQ, ASRQ, LSRQ, ROLQ, RORQ, INQ, DEQ.
    *   [X] Verify/Complete: MAP (0x5C stub), NEG (0x42), RTN (0x62).
    *   [X] EOM Prefix Logic (quad-flat): execute_from_mem() now detects $42 $42 $EA $base (quad-flat) vs $42 $42 $base (quad) and sets eom_prefix=2 directly; dispatch_table_t gained quad_eom[256] for routing.
*   [X] 45GS02 Architectural Features:
    *   [X] 16-bit Stack Pointer: `cpu_t.s` widened to `unsigned short`; CLE/SEE now functionally switch between emulation (E=1, page-1 stack) and extended (E=0, full 16-bit SP) modes. All 45GS02 stack operations are E-flag-aware. 45GS02 initialises with E=1 (emulation mode) matching real hardware reset behaviour.
    *   [X] E Flag (Extended Stack): Bit 5 of P toggles 16-bit vs 8-bit stack via CLE/SEE.
    *   [X] 32-bit Address Support: Implement the 45GS02's ability to access up to 1MB/4GB via MAP and EOM instructions.
*   [X] Decimal Mode Accuracy: Ensure ADC/SBC correctly handle the D flag across all CPU variants (6502 vs 65C02 differences). Shared do_adc()/do_sbc() helpers in cpu.h implement correct BCD arithmetic for all variants; N/V/Z flags are always well-defined (not emulating NMOS quirks).
*   [ ] Cycle Accuracy: Refine cycle counts for page crossings, branch success/failure, and 65CE02/45GS02 specific timing.
*   [ ] Processor-Specific Reset State: Different CPU variants have slightly different register states after RESET.
*   [ ] Complete Opcode Byte Values: Provide the correct machine code byte value for all opcodes in all processor tables (currently only partially implemented).
*   [X] Fix Branch Logic: Resolved double-increment bug in Bxx instructions for 6502 (opcodes_6502.c).
*   [X] Fix Branch Logic: Apply similar fixes to 65C02, 65CE02, 45GS02 opcode files.

2. Memory System Enhancements
----------------------------
*   [X] Simulator Refactoring: Converted to ROM-based execution model to support loops and backward jumps.
*   [X] Self-Modifying Code (SMC): Execution model switched from static ROM dispatch to execute_from_mem(), enabling programs to write to and execute from any address — including patching opcodes, operands, and jump targets at runtime. Default program origin moved from $0800 to $0200 (conventional placement above ZP and stack). Test coverage: smc_loop, smc_patch_jump, smc_patch_opcode, smc_store_operand.
*   [X] Label Resolution: Implemented correct label resolution for relative branches (BNE, BEQ, etc.).
*   [ ] Memory Mapping (MMU): Support for banking and the MEGA65 MAP instructions.
*   [ ] I/O Region Simulation: Implement basic hardware registers (e.g., VIC-II, SID, or MEGA65 specific registers) to allow running more complex programs.
*   [ ] Read/Write Callbacks: Allow attaching functional hooks to specific memory addresses (useful for simulating hardware I/O).
*   [ ] VIC Register Model: Maintain a structured parse of VIC-II ($D000–$D3FF), VIC-III ($D000–$D7FF), and VIC-IV ($D000–$DFFF) register banks so that the GUI graphics viewer can query video mode, sprite state, color RAM, and bitmap/character base addresses without raw memory scanning.
*   [ ] Memory Protection: Add the ability to mark regions as Read-Only or No-Execute.

3. Debugging & Trace Features
----------------------------
*   [X] Support standard pseudo-ops: `.org`, `.byte`, `.word`, `.text`, `.align`.
*   [X] Add support for numeric constants in decimal (`123`) and binary (`%10101010`). Also support character contants such as 'a'. 
*   [X] Allow psuedo-op to include a binary at the current PC. (.bin "filename")). Also allow Interactive command "bload" to perform the same function using 'bload "filename" $addr'  where addr is the address to start loading at. 
*   [X] Allow binary file saving of mem (eg: BSave "filename" START END ) will save the filename at address range.
*   [ ] Allow disassembly of memory (eg: D or Disasm START END) if just D then use PC.
*   [ ] Improved Trace Formatting: Make trace output more readable and include register changes (e.g., "A: 00 -> 01").
*   [X] Conditional Breakpoints: Support breakpoints that trigger only when a condition is met (e.g., "PC == $1234 && A == $00").
*   [X] Reverse Debugging / Execution History: Full forward/backward time-travel via delta-log ring buffer.
    *   [X] History entry format: full cpu_t snapshot + sparse memory delta (up to 16 {addr, old_value} pairs per instruction from the write log in memory_t).
    *   [X] Ring buffer with large default depth (131072 entries, ~10 MB); automatic eviction of oldest entry when full. Capacity defined as SIM_HIST_DEFAULT_DEPTH in sim_api.h.
    *   [X] `sim_history_step_back()` / `sim_history_step_fwd()` API in sim_api.h; history pushed automatically inside sim_step() per instruction. All execution paths (step and run) record to history.
    *   [X] CLI commands: `stepback` / `sb` (step back one instruction) and `stepfwd` / `sf` (step forward); local 4096-entry ring buffer in commands.c mirrors the API design.
    *   [X] MCP tools: `step_back` and `step_forward` added to plugin-gemini/server.js.
    *   [ ] GUI time-machine pane: timeline slider, step-back/step-forward buttons, "rewind to breakpoint" (reverse-continue until a condition is true).
*   [ ] Stack Visualization: Add a dedicated view for the current stack contents and depth.
*   [ ] Disassembler Integration: Improve the trace and memory viewer to show disassembled instructions alongside hex values.

4. Performance & Infrastructure
------------------------------
*   [ ] JIT or Optimized Interpreter: Explore performance optimizations for long-running simulations.
*   [ ] Unit Test Suite: Create a comprehensive test suite for all opcodes, especially for edge cases like decimal mode and overflow flag.
*   [ ] Integration with External Tools: Support loading standard binary formats (e.g., .prg, .bin) and symbol files from common assemblers (ca65, acme).
*   [ ] CMake Support: Add a CMakeLists.txt for better cross-platform build support.

5. User Interface (CLI/Web/MCP)
--------------------------
*   [X] Interactive CLI Mode: A "monitor" style interface where users can type commands (step, continue, examine, modify) without restarting. (Required for MCP).
    *   [X] Interactive mode can now be entered without specifying a source file (`./sim6502 -I`); pressing Enter on a blank line steps one instruction.
    *   [X] Inline assembler: `asm [addr]` enters an assembly sub-mode at the given address (default: current PC); all pseudo-ops supported; exit with `.` alone on a line; assembled bytes echoed in monitor format.
*   [X] MCP Server Implementation: Wrap the simulator in a Model Context Protocol server to expose execution control to LLMs.
*   [ ] Web-based UI: Use the existing plugin-gemini structure or similar to create a visual representation of the CPU state and memory.
*   [ ] Configuration Files: Support for .ini or .json files to specify CPU type, memory maps, and pre-loaded binaries.
*   [ ] GUI — VIC Viewer pane: Live rendered video output derived from current VIC-II/III/IV register state and memory contents; sprite inspector, character set browser, bitmap viewer, color RAM editor. Export frames and sprites as PNG.
*   [ ] GUI — Execution History / Time Machine pane: Timeline slider over the delta-log ring buffer; step-back/step-forward buttons wired to sim_history_step_back()/sim_history_step_fwd(); table of historical register states; "rewind to breakpoint" reverse-continue.
*   [ ] GUI — Remote Hardware Integration:
    *   [ ] VICE remote monitor protocol (TCP, default port 6510): connect/disconnect UI; forward step/run/pause/memory-read/write/breakpoint commands to a live VICE instance; all display panes driven from remote state when connected.
    *   [ ] M65dbg serial protocol: connect to MEGA65 via USB serial (typically 2 Mbps); send debug commands to read/write 28-bit address space, set breakpoints, single-step; device path + baud rate configurable.
    *   [ ] M65dbg Ethernet protocol: connect to MEGA65 via IP address (port 4510); same command set as serial; enables debugging over LAN without a USB cable.

6. Documentation
---------------
*   [ ] API Documentation: Document the internal structures (cpu_t, memory_t) for developers.
*   [ ] Instruction Set Reference: Generate or provide a reference for the supported opcodes of each processor type.
*   [ ] Tutorial/Examples: Add more example programs demonstrating specific processor features (e.g., 45GS02 Z register, 65CE02 indirect addressing).

7. Interface Gaps (CLI vs MCP)
-----------------------------
*   [ ] **MCP Missing Features:**
    *   Symbol Table Support: No tools to load custom symbol files or use presets (e.g., C64, Mega65).
    *   [X] Breakpoints: Implemented `set_breakpoint`, `clear_breakpoint`, `list_breakpoints`, and `run_program` tools.
    *   [X] Execution History: `step_back` and `step_forward` tools added.
    *   Memory Statistics: No equivalent to the `-s` flag.
    *   Trace: No tool to enable/disable execution tracing or set a trace file.
    *   Interrupts: No tools to trigger IRQ or NMI.
*   [ ] **CLI Missing Features:**
    *   Dynamic Code Loading: CLI requires a file path; cannot accept raw assembly string arguments (unlike MCP's `load_program`).

8. Code Infrastructure
----------------------
*   [ ] Reduce Code Duplication: Use macros or generic addressing mode handlers to implement opcodes. Currently, many instructions (like `LDA`) have nearly identical implementations for every addressing mode.
*   [ ] **Modularization**:
    *   [ ]  Split the monolithic `sim6502.c` into separate modules: `assembler.c`, `debugger.c`, `cpu_dispatch.c`, and `main.c`.
    *   [ ] Decouple the interactive debug shell from the core execution logic.
*   [ ] **Memory Management**:
    *   [ ] Move large structures like `memory_t` from the stack to the heap to prevent potential stack overflows.
    *   [ ] Implement a proper cleanup function for `memory_t` to `free()` sparse memory pages allocated via `calloc`.
*   [ ] **Error Handling**: Improve error reporting during assembly parsing, providing line numbers and specific error messages for invalid syntax.
